# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Использован датасет `S12-hw-dataset.csv` с почасовыми значениями целевой переменной. Прогнозировалось значение на следующий час (горизонт = 1).
- Данные разделены хронологически на `train` (70%), `validation` (15%) и `test` (15%).
- Сравнены три базовые модели: `B1` – наивный прогноз (последнее известное значение), `B2` – скользящее среднее за 7 часов, `B3` – Ridge-регрессия на лаговых и скользящих признаках (lag_1, lag_7, lag_14, rolling_mean_7, rolling_std_7, день недели в виде синуса и косинуса).
- Также обучена рекуррентная нейросеть `R1` (GRU) на окне из 30 предыдущих наблюдений.
- Лучшей моделью на валидации оказалась `R1` (GRU), она же показала наилучшие результаты на тесте.

## 2. Среда и воспроизводимость

- Python: 3.13.9
- pandas / numpy / scikit-learn: версии, установленные в окружении, обеспечивают корректную работу
- torch: 2.5.1+cu121
- Устройство: GPU (CUDA)
- Seed: 42
- Как запустить: открыть `HW12.ipynb` и выполнить Run All.

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: почасовая (24 значения в сутки)
- Горизонт прогноза: 1 (следующий час)
- Размер датасета: 4320 строк
- Диапазон дат: 2025-01-01 00:00:00 – 2025-06-29 23:00:00
- Есть ли дополнительные внешние признаки: нет, используются только лаги и календарные признаки, построенные из времени.
- Комментарий: ряд демонстрирует выраженный тренд и сезонность, что видно на графике; пропуски отсутствуют, выбросов не наблюдается. Задача прогнозирования осложняется наличием суточных и недельных циклов, которые требуют учёта предыдущих значений.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Разбиение выполнено хронологически: первые 70% строк – `train`, следующие 15% – `validation`, последние 15% – `test`.
- Конкретные интервалы:  
  - train: 2025-01-01 00:00:00 – 2025-05-06 23:00:00 (3024 строки)  
  - val:   2025-05-07 00:00:00 – 2025-06-02 23:00:00 (648 строк)  
  - test:  2025-06-03 00:00:00 – 2025-06-29 23:00:00 (648 строк)
- Почему `random split` некорректен: случайное перемешивание нарушило бы временную структуру, модель могла бы использовать будущие данные для обучения, что не соответствует реальному сценарию прогнозирования. Временной split гарантирует, что обучение происходит только на прошлом, а валидация и тест – на будущем.

### 4.2. Признаки для baseline-моделей

- Лаги: `lag_1`, `lag_7`, `lag_14`.
- Rolling-признаки: `rolling_mean_7` (среднее за 7 предыдущих значений, без учёта текущего) и `rolling_std_7` (стандартное отклонение за 7 предыдущих значений).
- Календарные признаки: день недели преобразован в синус и косинус (`dow_sin`, `dow_cos`) для учёта циклического характера.
- Пропуски после построения лагов/rolling: отброшены строки с `NaN` (первые 14 строк для `lag_14` и `rolling`).
- Масштабирование: использован `StandardScaler`, обученный только на тренировочных данных. Валидация и тест преобразованы с помощью этого же scaler’а.

## 5. Модели и эксперименты (B1, B2, B3, R1)

- **B1 (`naive-last`):** прогноз = последнее известное значение (`lag_1`). Простейший базовый подход.
- **B2 (`moving-average`):** прогноз = скользящее среднее за последние 7 часов (`rolling_mean_7`). Сглаживает шум, но отстаёт от резких изменений.
- **B3 (`ridge-lag-features`):** линейная Ridge-регрессия на наборе признаков (лаги, rolling-статистики, день недели). Позволяет учесть больше информации из прошлого.
- **R1 (`gru-forecast`):** однослойная GRU с размером скрытого состояния 32, входное окно `window_size = 30`. Прогноз делается на один шаг вперёд. Используется MSE loss, оптимизатор Adam (lr=1e-3), обучение 50 эпох с сохранением лучшей модели по валидационной MSE.
- Основная метрика выбора лучшей модели: MAE (на валидации).
- Дополнительные метрики: RMSE, MAPE.
- Batch size: 32.
- Максимальное число эпох: 50 (ранняя остановка не применялась, но сохранялась лучшая модель).
- Optimizer: Adam.
- Learning rate: 1e-3.
- Сохранение лучшей модели: веса модели, показавшей минимальную валидационную MSE, сохранены в `artifacts/best_gru.pt`, а также сохранён конфиг модели в `best_gru_config.json`.

## 7. Анализ

Наивный baseline (B1) оказался на удивление сильным: его MAE = 6.46, тогда как B2 (moving average) проигрывает почти в два раза. Это указывает на то, что ряд во многом предсказуем по предыдущему значению, а усреднение вносит излишнее сглаживание и запаздывание. Лаговые и rolling-признаки (B3) позволили улучшить результат по сравнению с B2, но всё равно не превзошли наивный подход. Возможно, линейная модель не смогла эффективно использовать недельную сезонность, хотя признаки дня недели были добавлены.

GRU (R1) смогла превзойти наивный метод. Это объясняется способностью GRU учитывать долгосрочные зависимости в последовательности (окно 30) и адаптироваться к тренду и сезонности. Кривые обучения показывают, что модель не переобучилась: валидационная ошибка стабилизировалась на уровне ниже тренировочной, что говорит о хорошей обобщающей способности.

Тренд и сезонность в данных визуально заметны: ряд растёт с января по июнь, имеет суточные и недельные колебания. GRU, в отличие от линейной регрессии, смогла извлечь эти паттерны из данных, что и дало выигрыш в метриках.

Утечки данных могли бы возникнуть при использовании будущих значений для расчёта признаков (например, при неправильном сдвиге rolling-статистик). В работе все признаки строились только на прошлых данных (shift(1) для rolling). Также масштабирование проводилось только на тренировочных данных, что исключает использование информации из будущего.

Выбор по валидации, а не по тесту, принципиален, чтобы избежать подгонки под тестовую выборку. Валидационная выборка используется для выбора гиперпараметров и лучшей модели, а тест – для финальной непредвзятой оценки.

Наиболее показательные ошибки модели можно увидеть на графике прогноза: в основном GRU точно следует за фактическими значениями, но иногда ошибается в пиках и резких сменах тренда. Это типично для нейросетей на временных рядах.

## 8. Итоговый вывод

Как базовый подход для данной задачи логично использовать GRU, так как она дала наилучший результат. Однако, если требуется максимальная интерпретируемость и скорость, можно ограничиться наивным прогнозом, который оказался достаточно точным. Простые линейные модели на признаках (Ridge) в данном случае не принесли выигрыша.

Переход к GRU (или LSTM) имеет смысл, когда временной ряд содержит сложные сезонные зависимости и тренды, которые не улавливаются простыми линейными правилами. При этом важно иметь достаточно данных для обучения нейросети (в нашем случае ~3000 точек) и правильно организовать валидацию по времени.

Главный вывод про корректную валидацию временных рядов: разбиение должно быть хронологическим, иначе оценка модели будет необъективной, так как она может «подглядывать» в будущее. Любое построение признаков (лаги, скользящие средние, масштабирование) должно использовать только информацию, доступную на момент прогноза.
